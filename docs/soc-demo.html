Expected workflow

Manual browser path

  1. User opens the SkyBridge homepage.
  2. User views the Dublin to Paris flight results.
  3. User selects the Economy Saver option.
  4. User enters fake passenger details.
  5. User clicks Continue to Payment.
AI-assisted workflow

Comet browser path

  1. Comet reads the controlled page content.
  2. Comet follows navigation links across the site.
  3. Comet selects the requested flight option.
  4. Comet fills the form using fake test details.
  5. Comet reaches the controlled verification step.
SOC visibility

Evidence expected

  1. Sysmon records browser and process activity.
  2. Suricata records HTTP requests to SkyBridge pages.
  3. Velociraptor validates endpoint process evidence.
  4. Splunk indexes the events by sourcetype.
  5. T1-T6 searches support the investigation timeline.

Expected evidence matrix

The table below shows what each telemetry source is expected to contribute during the controlled demonstration.

Evidence source Expected evidence Related search/report
Sysmon Browser process activity, process timestamps and command-line data where available T1, T5
Suricata HTTP paths such as /index.html, /flights.html, /ticket.html, /inbox.html and /confirmation.html T2
Velociraptor pslist Running process evidence from the Windows endpoint T3
Velociraptor pstree Parent-child process context and additional endpoint process validation T4
Splunk Combined event timeline using endpoint, network and process evidence T6

Lab safety notice

This website is a controlled internal lab target. It does not process real bookings, payments, credentials or personal data.

The EICAR test file is used only as a harmless antivirus test file inside the lab. The real EICAR file is not included in the GitHub repository.

This demo page is designed to organise safe test scenarios and expected SOC evidence. It does not contain hidden malicious prompts, credential collection, or real phishing functionality.
